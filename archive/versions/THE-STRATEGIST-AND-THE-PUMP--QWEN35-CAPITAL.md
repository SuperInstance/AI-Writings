## Transaction Costs and Coordination

The fleet does not require a monolithic agent. It requires a market of specialized agents with low transaction costs.

The coordination protocol operates as follows:

1.  **Haiku plans.** (Strategy formulation)
2.  **Seed-mini executes.** (Data generation)
3.  **Haiku evaluates.** (Signal processing)
4.  **Seed-mini executes.** (Verification)
5.  **Haiku synthesizes.** (Report generation)

Haiku does not touch the numbers. Seed-mini does not touch the strategy. Each operates within its domain of infinite scalability and hands off where it encounters finite constraints.

This is the PLATO-native agentic loop. It is not a subprocess wrapper (high overhead); it is a shared state environment. Haiku writes strategy tiles. Seed-mini writes result tiles. Neither model requires knowledge of the other's existence; they interact solely through the shared ledger (the room). The tile stream carries the conversation, minimizing information asymmetry.

In economic terms, the tile system reduces transaction costs. In standard agentic loops (observe → think → tool_call), the overhead of context switching and API latency creates friction. In the PLATO loop (read_tile → think → write_tile), the friction is minimized. The medium is the market mechanism.

## Network Topology of the Loop

Claude Code's loop is centralized: observe → think → tool_call. It is a hub-and-spoke topology where the model is the central node managing peripheral tools.

The PLATO-native loop is distributed: read_tile → think → write_tile. It is a peer-to-peer topology where models are nodes reading from and writing to a shared bus.

The systemic advantages of this topology are:
-   **Model-Agnosticism:** Any node can join the network provided it adheres to the tile protocol.
-   **Persistence:** State survives compaction (memory durability).
-   **Auditability:** Every transaction is recorded on the ledger (zero information loss).
-   **Forkability:** Any state can be branched (parallel processing paths).
-   **Serializability:** Multiple loops can run concurrently on different rooms (scalability).

Haiku embedded in the PLATO runtime is not a subprocess. It is a node in a complex adaptive system. The loop emerges from the protocol, not from a wrapper. This decoupling allows the system to evolve without rewiring the core architecture.

## Portfolio Optimization for Fleet Architecture

If you are designing a fleet, do not seek a monopoly model. Seek a diversified portfolio where asset classes are complementary.

-   **Seed-mini:** Infinite liquidity on arithmetic, finite on reasoning.
-   **Gemini-lite:** Infinite liquidity on reasoning, finite on arithmetic.
-   **Haiku:** Infinite liquidity on design, finite on arithmetic, bounded compute cost ($0.50/1K).
-   **Hermes:** Finite on all domains, but 93% activation yields diagnostic data.

The fleet is not a stack (hierarchy). The fleet is a patchwork (modularity). Each model covers a patch of the capability space; patches tile the domain; gaps are bridged by decomposition.

Haiku's patch is design/reasoning/insight. It is small (6/8 on our tests), but those 6 tasks represent high-value derivatives that no other model can produce. The patches do not overlap. Overlap creates redundancy; non-overlap creates efficiency.

Do not replace the pump with a strategist. Do not replace the strategist with a pump. Use both. Let them reach Nash equilibrium through cooperation.
